What is WeSync?

WeSync is a platform developed to satisfy CSCI 331 Course project requirements. The functionality of WeSync could be compared to an academic version of LinkedIn, but with a twist. Users can message one-another, add courses, join groups in courses, message other users, but most importantly rate one another.The development process included following a series of UX Design stages which will be covered on this page.

Demo the platform
{# row 2 #}
The Problem

Learning to work in groups and collaborate is an important experience that most university experience. From all parts of academia from software to business to writing, working in groups to complete course projects is also incredibly common. However, most students find out quite quickly that the main challenge in group projects is not the assignment itself, but learning to work well with the teammates available. Furthermore, the current/recent pandemic has pushed much of this collaboration online, bringing new challenges such as networking. WeSync aims to help students connect with peers that they can then collaborate successfully with while also allowing students to be held accountable for their abilities to collaborate.

Tools used

During this project, an incredibly wide variety of tools and platforms were used to produce the final prototype available at the above 'demo' link. Github was used for version control and to set up task assignments. Adobe XD was used to create an interactive prototype of what we thought the platform should look like during our UX/UI design process. Django was the chosen web framework that was used to develop the demo. SQlite was used to save all the data for our models, such as users, messages, courses and multiple other database entities. Ajax was used to make our website more dynamic, including dynamic searches, and asynchronous updating of pages. Bootstrap was used for UI development, Docker was used as the projects development containerization service, and Google's Cloud Run was used as deployment (The demo available above is hosted on AWS, NOT Cloud Run.)

{# row 3 #}
The UX/UI Design process

The UX/UI Design process of WeSync included each of the following stages being followed.

  1. Empathize - Research your users
  2. Define - State Your user's needs and problems
  3. Ideate - Challenge assumptions and create ideas
  4. Prototype - Start to create solutions
  5. Test - Try your solutions out

Developing the database schema

One of the reasons Django was chosen as the web framework to use was that Django out of the box as an incredibly powerful object relational mapper, which essentially makes working with databases easier for object-orientated brains.

By defining our models, and therefore database schema using this ER diagram it made it way easier to then turn our ideas into concepts.

{# row 5 #}
Putting an engine in the truck

We had the gas (the databases) now we needed to build the engine of this web application. This included creating three django applications in the WeSync project, Course, User Profile, and Message. Each of these apps then were developed to be fully functional according to the model, template, view architecture.

The largest challenges of this stage were finding ways for the applications to communicate with one-another effectively. Using tools just as Ajax I was able to create a level of functionality that I'm actually quite proud of. During the development phase, there were many reality checks were hopes and dreams were tuned to be a bit more realistic with the development timeframe that we had (3 weeks).

{# row 6 #}
Cool it works on your laptop, how do I put it on the internet?

For deployment of our prototype we chose to create a custom docker image and to run that image on Google Cloud's Cloud Run platform. This service was chosen from the many because of its support of containerized applications and expandability. Additionally, as a developer I liked the idea that in the future if need be, Cloud Run would interface with other Google products very well.

It is important to note that the demo available at the link above is not hosted on Google Cloud's Cloud Run platform. Instead, a simplified version of the site was moved to AWS's Beanstalk hosting service for affordability purposes. Because it is only a lightweight demo there was no need to pay for Google's more expensive services.
